# i18n — translating the FabMo UI

FabMo supports per-machine localization via key-based JSON dictionaries.
English is the source of truth; other languages mirror its structure.

## Files

- `i18n/en.json` — English strings, source of truth. Edit this when
  adding or changing user-facing copy.
- `i18n/<code>.json` — translations. New languages are added by
  dropping a `<code>.json` here. `_meta.language` gives the display
  name shown in the language selector.
- `i18n/index.js` — server-side helpers: `t(key, lang, vars)`,
  `getDict(lang)`, `listLanguages()`, `reload()`.
- `i18n/errors.js` — tagged-error helpers (see below).
- `dashboard/static/js/libs/i18n.js` — client-side: `window.t(key, vars)`,
  `window.i18nApply(root)`, `window.i18nReady` (Promise).
- `routes/i18n.js` — exposes `/i18n/languages` and `/i18n/dict/:lang`.
- `scripts/i18n-export.js` / `scripts/i18n-import.js` — CSV roundtrip
  for handing off to translators or AI.

## Active language

Stored on `config.engine.language` (default `"en"`). Change via the
standard config POST:

```bash
curl -X POST -H 'Content-Type: application/json' \
  -d '{"engine":{"language":"ja"}}' http://<host>/config
```

Dashboard fetches the active dict on page load. Reload the browser
after switching.

## Adding a translatable string

### In HTML

Mark the element with `data-i18n="<key>"` (for text content) or
`data-i18n-{title,placeholder,aria-label,value}="<key>"` for the
matching attribute. Leave the English text as the fallback so no-JS
and pre-fetch renders still read correctly:

```html
<span data-i18n="keypad.go_to">Go To</span>
<button data-i18n-title="keypad.go_to_tooltip" title="Move to...">…</button>
```

Then add the English string to `i18n/en.json`:

```json
{
  "keypad": { "go_to": "Go To", "go_to_tooltip": "Move to..." }
}
```

The walker runs once after the dict loads (`i18nReady.then(...)`).
For dynamically-injected DOM, call `window.i18nApply(rootElement)`
after insertion.

### In JS

```js
window.i18nReady.then(function () {
    el.textContent = window.t("toolbox.cut");
});
```

Or for vars:

```js
window.t("toolbox.cut_summary", { diameter: 0.5 });
// in en.json: "cut_summary": "Bored {diameter}\" hole"
```

### In server code (Node)

```js
var i18n = require("./i18n");
log.info(i18n.t("startup.ready", "en"));
```

## Error messages

Use `i18nError(code, vars)` instead of `new Error("...")` for errors
that surface to the user:

```js
var { i18nError } = require("./i18n/errors");

if (machine.state === "manual") {
    throw i18nError("errors.machine.cannot_arm_from_state", {
        action: actionType,
        state: machine.state,
    });
}
```

At the response boundary, translate to the caller's language:

```js
var { errorMessage } = require("./i18n/errors");

try { ... } catch (e) {
    var lang = config.engine.get("language") || "en";
    res.json({ status: "fail", message: errorMessage(e, lang) });
}
```

Plain `new Error("...")` continues to work — `errorMessage` falls
back to `e.message`. Migration can be done incrementally.

## Bundled apps (.fma)

Each `.fma` app runs in its own iframe with a fresh window context, so
the parent dashboard's `window.t` isn't reachable. Apps opt in by
copying `i18n.js` into their `js/` directory and including it like
any other library script:

```html
<script src="js/lib/jquery.js"></script>
<script src="js/i18n.js"></script>      <!-- before other app scripts -->
```

The script fetches `/i18n/dict/<lang>` (route is registered at the
root, no version-hash redirect interference) and exposes the same
`window.t`, `window.i18nApply`, and `window.i18nReady` as the parent.

After including: mark strings in the app's HTML with `data-i18n` and
`data-i18n-{title,placeholder,aria-label,value}` attributes exactly
like the dashboard chrome. Add the English copy to `i18n/en.json`
under a per-app namespace (`sb4.*`, `previewer.*`, etc.).

**Reference implementation:** `dashboard/apps/sb4.fma/`. The top
navigation, workflow panel, and first two macro buttons are migrated
as a worked example. Replicate the pattern through the rest of sb4
and the other apps (`previewer.fma`, `configuration.fma`,
`macro_manager.fma`, …) one section at a time.

When you copy `i18n.js` into a new app:

```bash
cp dashboard/static/js/libs/i18n.js dashboard/apps/<app>.fma/js/i18n.js
```

Keep them in sync if the helper changes — they're literally identical
copies. (A future cleanup could publish a single canonical URL via a
static-mount route to drop the duplication.)

## Translator workflow

1. Export current keys + any existing target translations:
   ```bash
   node scripts/i18n-export.js ja > i18n/ja.csv
   ```
2. Hand the CSV to a translator, or feed it to an AI for first-pass
   translation. Three columns: `key`, `english`, `ja`. Fill in the
   third column.
3. Import the edited CSV:
   ```bash
   node scripts/i18n-import.js ja i18n/ja.csv
   ```
   Rows with an empty target are skipped so the JSON doesn't contain
   empty strings that would mask the English fallback.
4. Restart fabmo (or call `i18n.reload()` programmatically) to pick
   up the new dictionary.

## Adding a new language

1. Create `i18n/<code>.json` with `{ "_meta": { "language": "...",
   "code": "..." } }`.
2. Export keys to CSV, translate, import (above).
3. The language appears automatically in `/i18n/languages` and is
   selectable via `engine.language`.

## Conventions

- Keys use nested dotted paths grouped by area: `keypad.*`,
  `toolbox.*`, `errors.machine.*`. Group structure mirrors UI surfaces
  so translators can work section by section.
- Mark whole visible strings, not fragments — keep markup outside the
  translated value where possible. Translators don't need to deal with
  HTML.
- Plurals: not currently handled. If/when needed, add `tn(key, count,
  vars)` that selects between `<key>_one` and `<key>_other` (or
  similar). Japanese doesn't pluralize so the default is fine for now.
- Untranslated keys silently fall back to English. To find untranslated
  copy at runtime, view-source for keys that look like dotted paths
  rather than visible text.
