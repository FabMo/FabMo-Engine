# i18n — translating the FabMo UI

FabMo supports per-machine localization via key-based JSON dictionaries.
English is the source of truth; other languages mirror its structure.

## Files

- `i18n/en.json` — English strings, source of truth. Edit this when
  adding or changing user-facing copy.
- `i18n/<code>.json` — translations. New languages are added by
  dropping a `<code>.json` here. `_meta.language` gives the display
  name shown in the language selector.
- `i18n/index.js` — server-side helpers: `t(key, lang, vars)`,
  `getDict(lang)`, `listLanguages()`, `reload()`.
- `i18n/errors.js` — tagged-error helpers (see below).
- `dashboard/static/js/libs/i18n.js` — client-side: `window.t(key, vars)`,
  `window.i18nApply(root)`, `window.i18nReady` (Promise).
- `routes/i18n.js` — exposes `/i18n/languages` and `/i18n/dict/:lang`.
- `scripts/i18n-export.js` / `scripts/i18n-import.js` — CSV roundtrip
  for handing off to translators or AI.

## Active language

Stored on `config.engine.language` (default `"en"`). Change via the
standard config POST:

```bash
curl -X POST -H 'Content-Type: application/json' \
  -d '{"engine":{"language":"ja"}}' http://<host>/config
```

Dashboard fetches the active dict on page load. Reload the browser
after switching.

## Adding a translatable string

### In HTML

Mark the element with `data-i18n="<key>"` (for text content) or
`data-i18n-{title,placeholder,aria-label,value}="<key>"` for the
matching attribute. Leave the English text as the fallback so no-JS
and pre-fetch renders still read correctly:

```html
<span data-i18n="keypad.go_to">Go To</span>
<button data-i18n-title="keypad.go_to_tooltip" title="Move to...">…</button>
```

Then add the English string to `i18n/en.json`:

```json
{
  "keypad": { "go_to": "Go To", "go_to_tooltip": "Move to..." }
}
```

The walker runs once after the dict loads (`i18nReady.then(...)`).
For dynamically-injected DOM, call `window.i18nApply(rootElement)`
after insertion.

### In JS

```js
window.i18nReady.then(function () {
    el.textContent = window.t("toolbox.cut");
});
```

Or for vars:

```js
window.t("toolbox.cut_summary", { diameter: 0.5 });
// in en.json: "cut_summary": "Bored {diameter}\" hole"
```

### In server code (Node)

```js
var i18n = require("./i18n");
log.info(i18n.t("startup.ready", "en"));
```

## Error messages

Use `i18nError(code, vars)` instead of `new Error("...")` for errors
that surface to the user:

```js
var { i18nError } = require("./i18n/errors");

if (machine.state === "manual") {
    throw i18nError("errors.machine.cannot_arm_from_state", {
        action: actionType,
        state: machine.state,
    });
}
```

At the response boundary, translate to the caller's language:

```js
var { errorMessage } = require("./i18n/errors");

try { ... } catch (e) {
    var lang = config.engine.get("language") || "en";
    res.json({ status: "fail", message: errorMessage(e, lang) });
}
```

Plain `new Error("...")` continues to work — `errorMessage` falls
back to `e.message`. Migration can be done incrementally.

## Third-party apps

Apps not in this repo (uploaded by users, shipped by partners) opt into
translation by following this convention:

### File layout

```
my-app.fma/
├── package.json          ({"name": "my-app", "version": "1.0.0", ...})
├── i18n/
│   ├── en.json           (source of truth, English)
│   ├── ja.json           (translation, optional)
│   └── ja.meta.json      (auto-written by i18n-import; do not edit)
├── js/
│   └── i18n.js           (copy of /dashboard/static/js/libs/i18n.js)
├── index.html            (uses data-i18n attributes)
└── ... other app files ...
```

### Key namespacing

Every key in the app's `en.json` must start with the app's id (taken
from `i18n/en.json` `_meta.app`, or `package.json` `name`, or the
directory name minus `.fma`, in that order). The server merges all app
dicts into one global dict at startup — namespacing is the only
collision protection.

```json
{
    "_meta": { "app": "my-app", "language": "English", "code": "en" },
    "my-app": {
        "title": "My App",
        "buttons": { "go": "Go" }
    }
}
```

### Server-side auto-merge

On startup, `i18n/index.js` walks `dashboard/apps/*.fma/i18n/` AND
`/opt/fabmo/approot/approot/*.fma/i18n/` and deep-merges every
`<code>.json` into the dict for that language. The merged dict is
served at the existing `/i18n/dict/<code>` endpoint — no changes
needed for clients. Conflict policy: later writes override (so any
core entry wins over an app entry that mistakenly uses the same key).

Server log on startup shows what was merged:
```
i18n: merged my-app.fma/en.json [namespaces: my-app]
i18n: merged my-app.fma/ja.json [namespaces: my-app]
```

### Pre-submission analyzer

Developers run this before submitting an app for translation review:

```bash
node scripts/i18n-analyze.js path/to/my-app.fma
```

The analyzer reports:
- **App** id and namespace it derived
- **Dictionary** stats: total keys, longest values
- **Translations** present and their coverage percent vs en.json
- **Issues** (blockers): keys outside the app's namespace, malformed
  JSON, etc.
- **Warnings**: empty values, suspicious entries
- **Coverage scan**: heuristic walk of `.html` / `.js` for user-visible
  English text that has no `data-i18n` / `t()` marking. Reports
  candidates with file:line. Caveat — multi-line tags and dynamically-
  built strings may produce false positives or false negatives.

To generate a translator-ready CSV for a specific language:
```bash
node scripts/i18n-analyze.js path/to/my-app.fma --export ja > ja.csv
```

Same 4-column format as the core export (`key, english, ja, status`).

`--json` for machine-readable output (CI integration etc.).

### Submission flow

1. Developer adds `i18n/en.json` + `data-i18n` markup; copies
   `i18n.js` into `js/`.
2. Runs `i18n-analyze` until issues are zero and coverage candidates
   are reviewed.
3. Optionally pre-translates one language via AI as a starting point.
4. Submits the `.fma` directory.
5. FabMo team (or reseller's translator) runs `i18n-analyze --export
   <lang>` to get a CSV, fills in translations, runs `i18n-import
   <lang>` to commit `<lang>.json` + `<lang>.meta.json` back into the
   app directory.
6. App ships with its translations baked in. Installs cleanly on any
   FabMo regardless of operator language.

### Reference implementation

`dashboard/apps/hello-i18n.fma/` — minimal sample app with full i18n
setup. Read the source as a template for new apps.

## Bundled apps (.fma)

Each `.fma` app runs in its own iframe with a fresh window context, so
the parent dashboard's `window.t` isn't reachable. Apps opt in by
copying `i18n.js` into their `js/` directory and including it like
any other library script:

```html
<script src="js/lib/jquery.js"></script>
<script src="js/i18n.js"></script>      <!-- before other app scripts -->
```

The script fetches `/i18n/dict/<lang>` (route is registered at the
root, no version-hash redirect interference) and exposes the same
`window.t`, `window.i18nApply`, and `window.i18nReady` as the parent.

After including: mark strings in the app's HTML with `data-i18n` and
`data-i18n-{title,placeholder,aria-label,value}` attributes exactly
like the dashboard chrome. Add the English copy to `i18n/en.json`
under a per-app namespace (`sb4.*`, `previewer.*`, etc.).

**Reference implementation:** `dashboard/apps/sb4.fma/`. The top
navigation, workflow panel, and first two macro buttons are migrated
as a worked example. Replicate the pattern through the rest of sb4
and the other apps (`previewer.fma`, `configuration.fma`,
`macro_manager.fma`, …) one section at a time.

When you copy `i18n.js` into a new app:

```bash
cp dashboard/static/js/libs/i18n.js dashboard/apps/<app>.fma/js/i18n.js
```

Keep them in sync if the helper changes — they're literally identical
copies. (A future cleanup could publish a single canonical URL via a
static-mount route to drop the duplication.)

## Translator workflow

1. Export current keys + any existing target translations:
   ```bash
   node scripts/i18n-export.js ja i18n/ja.csv
   ```
   The CSV has four columns: `key`, `english`, `<lang>`, `status`.
   Status is one of:
   - **ok** — translation matches the English it was made from
   - **new** — no translation yet (empty target cell)
   - **stale (was: "…")** — English has changed since the translation
     was recorded; the old English text is shown so the translator
     can judge whether their translation still applies
   - **untracked** — translation exists with no sidecar record;
     happens for hand-edited JSON or pre-staleness-system entries.
     Round-tripping through import promotes these to "ok"
   - **orphaned** — translation exists but the key has been removed
     from `en.json`. The row can be deleted, or left in case the
     key returns

2. Hand the CSV to a translator, or feed it to an AI. Translator
   filters/sorts by the `status` column and focuses on `new` + `stale`.
   They only edit column 3.

3. Import the edited CSV:
   ```bash
   node scripts/i18n-import.js ja i18n/ja.csv
   ```
   Writes two files:
   - `i18n/ja.json` — runtime dict
   - `i18n/ja.meta.json` — sidecar recording the English text each
     translation was made against. Used by the next export to detect
     drift. Commit both files.

   Rows with an empty target are skipped so the JSON doesn't contain
   empty strings that would mask the English fallback.

4. Restart fabmo (or call `i18n.reload()` programmatically) to pick
   up the new dictionary.

## Detecting orphan keys in code

Keys in `en.json` that no longer have a static reference anywhere
(`data-i18n="key"`, `t("key")`, `i18n.t("key")`, `i18nError("key")`):

```bash
node scripts/i18n-orphans.js
```

Lists candidates for deletion. Caveat: dynamically-built keys (e.g.
`t("foo." + name)` or `tool.labelKey` field references) won't be
detected — review the list before deleting anything from `en.json`.

`--json` for machine-readable output.

## Release-time checklist

When making a release after English copy changes:

```bash
node scripts/i18n-export.js ja i18n/ja.csv   # see status summary on stderr
node scripts/i18n-orphans.js                  # candidates for cleanup
```

If status counts show `new` or `stale` > 0: send `ja.csv` through
the translator, re-import, commit `ja.json` + `ja.meta.json`. Repeat
for each `<lang>.json` you ship.

## Adding a new language

1. Create `i18n/<code>.json` with `{ "_meta": { "language": "...",
   "code": "..." } }`.
2. Export keys to CSV, translate, import (above).
3. The language appears automatically in `/i18n/languages` and is
   selectable via `engine.language`.

## Conventions

- Keys use nested dotted paths grouped by area: `keypad.*`,
  `toolbox.*`, `errors.machine.*`. Group structure mirrors UI surfaces
  so translators can work section by section.
- Mark whole visible strings, not fragments — keep markup outside the
  translated value where possible. Translators don't need to deal with
  HTML.
- Plurals: not currently handled. If/when needed, add `tn(key, count,
  vars)` that selects between `<key>_one` and `<key>_other` (or
  similar). Japanese doesn't pluralize so the default is fine for now.
- Untranslated keys silently fall back to English. To find untranslated
  copy at runtime, view-source for keys that look like dotted paths
  rather than visible text.
